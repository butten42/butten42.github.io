原型继承

Veröffentlicht am 2017-10-26

在做腾讯NEXT学位项目（打飞机）的时候，我期望定义的子类仅仅继承父类的个别属性，于是在最顶上的原型里添加了超多的参数，到具体的子类时我仅仅call他的部分属性，我以为这样可行，可是实际上子类里仍然保有全部的父类属性，在后续的传参数时因为子类的参数少了（因为子类的属性少）导致个别属性undefined。
于是发现自己在原型继承上还是一团浆糊。

定义

原型和原型链

每个对象都会在其内部初始化一个属性，就是prototype(原型)，当我们访问一个对象的属性时，
如果这个对象内部不存在这个属性，那么他就会去prototype里找这个属性，这个prototype又会有自己的prototype，
于是就这样一直找下去，也就是我们平时所说的原型链的概念。

new 运算符是如何工作的？
我看到了很多的解释
在MDN上的解释

当代码 new foo(…) 执行时：
一个新对象被创建。它继承自foo.prototype.
构造函数 foo 被执行。执行的时候，相应的传参会被传入，同时上下文(this)会被指定为这个新实例。new foo 等同于 new foo(), 只能用在不传递任何参数的情况。
如果构造函数返回了一个“对象”，那么这个对象会取代整个new出来的结果。如果构造函数没有返回对象，那么new出来的结果为步骤1创建的对象。(一般情况下构造函数不返回任何值，不过用户如果想覆盖这个返回值，可以自己选择返回一个普通对象来覆盖。当然，返回数组也会覆盖，因为数组也是对象。)

在一般解释中

1、创建一个空对象，并且 this 变量引用该对象，同时还继承了该函数的原型。
  2、属性和方法被加入到 this 引用的对象中。
3、新创建的对象由 this 所引用，并且最后隐式的返回 this 。

井字棋与极大极小算法

Veröffentlicht am 2017-10-19

FCC上的一个利用经典的极大极小算法的项目，井字棋（我家那边叫“三笔直”）.
规则很简单：

  • 游戏一共两人，轮流执子
  • 每回合就是在一个九格正方形里放置一子
  • 当任意一方摆出的子连成一线时获胜。

（所以最快三个会合可以重来一局，小时候和小伙伴们在墙根底下经常随时取地上的石子玩耍）
就结构和样式其实很简单啦，难点在于如何模拟AI的动作，
这时就可以参考最小最大算法了，我直接在网上找相关的文章，解释的比较好的就是知乎上的，我直接引用了:

先来说极小极大算法主要应用于什么样的游戏： 1. 零和游戏（Zero-sum Game）：意思就是你死我活，一方的胜利代表另一方的失败，比如，象棋，五子棋等。 2. 完全信息（Perfect Information）：玩家知道之前所有的步骤。象棋就是完全信息，因为玩家是交替着落子，且之前的步骤都能在棋盘上体现，但是石头剪子布就不是。这样的游戏通常可以把他们看作一个树状图，把每一种可能性列出来。比如下面这个井字棋游戏，Max代表你自己，Min代表你的对手。
这个时候我们需要给每一种结果一个分数，就是这里的Utility。这个分数是站在我自己（也就是Max）的角度评估的，比如上图中我赢了就是＋1，输了是－1，平局时0。所以，我希望最大化这个分数，而我的对手希望最小化这个分数。（在游戏中，这个分数被称为static value。）这里要说一下，井字棋是个比较简单的游戏，所以可以列出所有可能的结果。但是，大部分游戏是不太可能把所有结果都列出来的。根据计算机运算量，我们可能只能往前推7，8步，所以这个时候分数就不只－1，1，0这么简单了，会有专门的算法来根据当前结果给不同的分数。

如何在乘坐交通工具时利用时间

Veröffentlicht am 2017-10-06

中秋回了趟家，本来打算在家看看书，做点有意义的事（话说简历也一直没改，爬虫也没做），然而一回家就感觉自己脑死了，连续两天，除了头天的晚上和朋友见了面，其它时间都缩在家里看番，活脱脱一个neet。
到了今天已经是离开家之前的最后一个晚上了，我觉得至少还是要写些东西才好。

如何在乘坐交通工具时利用时间

回家，不可避免的就要遭遇地理位移，在长途旅行里，我要怎么消磨呢？这个题目我本来是想写成“如何在火车公交上打发时间”，“如何在旅途中消磨时间”，“在公共交通上如何利用时间”等等，但是最后我还是觉得最后一个更合适一些。
我自己的话会有这么几种方法：

  • 听歌/播客/相声
  • 看书
  • 看电影
  • 看风景（发呆）
  • 看车内环境
  • 找人聊天，游戏
  • 睡觉
  • 玩手机（刷微信，知乎）

这里面最有价值的就是读书了，但是公共交通玩玩不太安静，而且列车地铁还好巴士的话时有行路陡峭，况且你也不一定有位置，所以并不适合看书。我几乎每次出门都会带上kindle结果真正看过的次数屈指可数。
我经常做的事就是听播客了，耗电量少，只要带耳朵就可以了，人还能假寐一会，就好像这次回家我一路上听了好几期的机核广播。不过缺点是公共交通上太吵了，你的音量就相对的要放大一些，对耳朵不太好。
看电影很不错啊，就是很容易没电。
看风景的时候一般和睡觉搭配。
如果身边有我想要搭讪的人也许会聊聊天吧，但是最近的我实在是没有什么心情和陌生人聊天。都是可遇不可求的事。
最后就玩手机了，但是长途旅行一般会遭遇电量耗尽这回事。我一直没有挂充电宝的习惯，只能是克制自己了，况且养成一个碎片化的阅读习惯的并不是好事。
有时候我也会拿纸笔写些东西，但是这在交通拥挤的时候基本不可能。

结论

没有特别好的方法来度过交通时间。候鸟不好当。有条件的话我希望能找到一个时间利用率高的方法，比如做飞机，动车高铁，人少一些，干扰少一些，但是公共交通嘛，总是不可避免的会被干扰。但是自己开车的话太辛苦。我们需要社会，同时厌恶着社会。
对了，要是附近有小孩子的话要尽量远离，在车上我几乎没有见过一个和可爱沾边的孩子。

vue 和knockout的最小实例

Veröffentlicht am 2017-09-25

关于knockout和xue的对比xue的官方guide上总结了这么一段话：

Knockout 是 MVVM 领域内的先驱，并且追踪依赖。它的响应系统和 Vue 也很相似。它在浏览器支持以及其他方面的表现也是让人印象深刻的。它最低能支持到 IE6，而 Vue 最低只能支持到 IE9。
随着时间的推移，Knockout 的发展已有所放缓，并且略显有点老旧了。比如，它的组件系统缺少完备的生命周期事件方法，尽管这些在现在是非常常见的。以及相比于 Vue 调用子组件的接口它的方法显得有点笨重。
如果你有兴趣研究，你还会发现二者在接口设计的理念上是不同的。这可以通过各自创建的 simple Todo List 体现出来。或许有点主观，但是很多人认为 Vue 的 API 接口更简单结构更优雅。

这次我通过一个功能相同的最小例子来探究一下两者的不同之处。

vue

knockout

比较

在这个小例子里，功能仅仅为：

  • 在视图上显示数据
  • 在视图上修改数据同时改变视图（双向绑定）
  • 在视图上输出计算结果
  • 在视图上绑定事件

如何在视图上绑定/显示数据？
knockout：HTML中使用data-bind=&quot;prop: prop-name&quot; 在js里新建vm类，并绑定ko.applyBindings(new VM())
vue：HTML中的指定作用位置里添加NaN 在js里新建vm实例，指明作用节点el（使用css节点）在data中建立属性（数组）
如何在视图上改变数据？
knockout: 首先数据必须是一个观察对象ko.obserable()在HTML里使用input标签绑定对象的value值
vue: 同样是input 标签使用v-model指令绑定对应属性
如何输出计算结果？
knockout:同样的所有元素必须是knockout中的观察对象,作为一个方法在原型里被定义
vue: 是一样的，不过在vue里这是一个被传入的对象属性
如何绑定事件？
Knockout:在对应的标签元素上绑定click: eventevent本身是一个方法
Vue:在对应的标签元素上绑定 v-on:click=&quot;event&quot; event包含在methods内。

总结

两者在视图上的绑定大体相似，在VM中Vue指定对象类型的方式更加清晰一些，在vue实例内的统一全是观察对象
